## 1. Формализованная логика расчёта

### Исходные данные

Файл `bank_data.xlsx`: 20 банков × 73 даты (2020-01-01 … 2026-01-01), помесячно. Колонки:
- Балансовые (срез на 1-е число месяца): `Assets`, `Loans_Total_Net`, `Loans_LLP`, `Equity`, `Client_Deposits`
- Потоковые: `Net_Income_Current` (помесячно), `Net_Interest_Income` (только Январь, Апрель, Июль, Октябрь)

### Обозначения

- `t` — отчётная дата (1-е число месяца)
- `NIC(t)` — Net_Income_Current на дату t
- `NII(t)` — Net_Interest_Income на дату t

### Про потоковые величины

В ТЗ про обе потоковые величины написано отражается «накопленным итогом», но период накопления у каждого коэффициента свой:

- ROE 12m — годовой коэффициент, прибыль аккумулируется за год (YTD, year-to-date). Цитата из ТЗ: «Рентабельность капитала **за год** … прибыль **за последние 12 месяцев** … потоковая величина, отражается накопленным итогом».
- LoanYield 3m — квартальный коэффициент, ЧПД аккумулируется внутри квартала (QTD, quarter-to-date). Цитата из ТЗ: «Доходность кредитного портфеля **за квартал** … ЧПД **за квартал** … потоковая величина, отражается накопленным итогом».

**NIC — YTD, соответствует ТЗ:**
Ratio-тест `Σ NIC за 12 месяцев / NIC(1 янв следующего года)` даёт 5–10 по всем банкам. Этот тест сам по себе не различает гипотезы: под YTD с ~равными месячными потоками ratio ≈ 6.5; под flow с ростом от года к году (реальный случай, 2020 → 2025 прибыль растёт в ~5×) ratio тоже может быть 7–10. Решающим аргументом стала реалистичность ROE: при YTD медиана 15% (соответствует банковской норме 10–20%), при flow подходе  медиана 87% с max 1217% (нереалистично)

**NII — QTD, соответствует ТЗ:**
Каждое значение NII на 1 Apr / 1 Jul / 1 Oct / 1 Jan — накопление ЧПД внутри соответствующего квартала, а не с начала года. В таком чтении формула `4 × NII(t) / loans_avg_Q(t)` прямо соответствует тексту ТЗ. Данные подтверждают: значения внутри года немонотонны (bank_1 2024: Q1=17 625 → H1=16 920), что возможно при QTD (меньшая прибыль следующего квартала), но невозможно при YTD. При QTD LoanYield ≈ 7% (соответствует рыночным), при YTD ≈ 0.9% (нереалистично). 

**LLM с ходу посчитала «накопленным итогом» и NIC, и NII - это было ошибкой чтения ТЗ.** Правильное прочтение (QTD для NII) снимае противоречие, которое видит LLM, между ТЗ и данными. Подробнее — в §4.

### Восстановление месячной прибыли из YTD

```
profit_month(t) =
  NIC(t),                если month(t) = 2   (январский поток = YTD на 1 фев)
  NIC(t) − NIC(t−1мес),  иначе               (включая t = 1 янв: FY − YTD на 1 дек)
```

NIC(1 янв Y) трактуется как FY(Y−1), т.е. YTD для нового года стартует с 0.

### Коэффициент 1: ROE 12m

```
profit_12m(t) = Σ profit_month(s), s ∈ {t−11, …, t}     # 12 месячных потоков
equity_avg(t) = mean(Equity(s)),   s ∈ {t−12, …, t}     # 13 балансовых точек

ROE_12m(t) = profit_12m(t) / equity_avg(t)
```

Условия расчёта:
- Требуется ≥13 точек истории (окно t−12 … t)
- Если хотя бы одно Equity(s) < 0 в окне → ROE не определён (по ТЗ)
- Если в окне есть пропуски NIC или Equity → ROE не определён

### Коэффициент 2: Доходность кредитного портфеля 3m

Определён только для квартальных дат: t ∈ {1 янв, 1 апр, 1 июл, 1 окт}.

```
loans_avg_Q(t) = mean(Loans_Total_Net(s)),  s — 4 точки квартала

LoanYield_3m(t) = 4 × NII(t) / loans_avg_Q(t)     # аннуализация ×4
```

NII(t) — QTD: накопление ЧПД внутри квартала, заканчивающегося на дату t (см. обоснование выше).

Условия расчёта:
- NII(t) определён (не NaN)
- Все Loans_Total_Net в окне ≥ 0
- loans_avg_Q > 0

### Коэффициент 3: LDR (Loans-to-Deposits Ratio)

```
Loans_Gross(t) = Loans_Total_Net(t) − Loans_LLP(t)     # LLP < 0, поэтому Gross > Net

LDR(t) = Loans_Gross(t) / Client_Deposits(t)
```

Условия расчёта:
- Loans_Gross(t) ≥ 0
- Client_Deposits(t) > 0

## 3. Найденные проблемы

### 3.1. Ошибки и аномалии в данных

В `bank_data.xlsx` синтетические данные, в которых возможны ошибки. Поэтому ни одна интерпретация не даёт идеальной картины. Конечный выбор делал я на основе анализа реалистичности получившихся коэффициентов, так как формальные проверки в таких случаях не дают однозначного прочтения. Иными словами, выбирался подход, при котором коэффициенты выглядели наиболее правдоподобно в большинстве случаев. 


| Проблема | Число строк | Затронутые банки |
|---|---|---|
| Equity < 0 | 172 | bank_3 (72/73), bank_11 (73/73), и др. |
| Loans_Total_Net < 0 | единичные | bank_1 (2023-07) |
| Client_Deposits < 0 | 5 | — |
| Assets < 0 | 3 | — |
| Текстовые значения с единицами | ~30 ячеек | «729 873,3 млн руб.», «… тыс. руб.» |
| Смешанные единицы измерения | — | млн руб. (основные), тыс. руб., млрд руб. |

### 3.2. Неоднозначности в методологии

1. **Скоуп «накопленного итога» у NII.** Фраза «ЧПД — потоковая величина, отражается накопленным итогом» в ТЗ к квартальному коэффициенту LoanYield 3m потенциально допускает два прочтения (разные инстансы LLM увидели неправильное прочтение): YTD (с начала года) или QTD (внутри квартала). Текст ТЗ («ЧПД **за квартал** ... отражается накопленным итогом») указывает на QTD, данные с этим согласуются (монотонность внутри года не требуется и действительно нарушается), а QTD даёт реалистичный LoanYield ≈ 7%. LLM предложила интерпретацию YTD для обеих потоковых величин, это ошибка чтения ТЗ (см. §4).

2. **NIC(1 янв Y): FY(Y−1) или ноль, начиная с нового года?** По значениям (bank_1 2020-01: 7 873, 2021-01: 8 989) — полная годовая прибыль предыдущего года, не сброс. Влияет на восстановление месячных потоков.

3. **Loans Gross: Net − LLP или Net как есть?** ТЗ говорит «кредиты (gross)», но отдельной колонки нет. Принята формула Gross = Net − LLP (LLP отрицательный → Gross > Net). Альтернатива — использовать Net.

4. **Среднее Equity для ROE: 13 точек, 12 точек или 2?** «Среднегодовое значение» допускает три трактовки: среднее 13 месячных балансов (t−12 … t), среднее 12 балансов (t−11 … t), или (Equity(t−12) + Equity(t)) / 2. Выбран вариант с 13 точками. Дельта между методами по всем 1031 валидной строке (18 банков, 2021-01 … 2026-01):

   | Метод | Медиана ROE | Среднее ROE | Медианная \|ΔROE\| vs 13pt | p95 \|ΔROE\| vs 13pt |
   |---|---|---|---|---|
   | 13 точек (принято) | 14.97% | 22.61% | — | — |
   | 12 точек | 14.85% | 22.02% | 0.07 пп | 2.3 пп |
   | (begin+end)/2 | 14.85% | 20.78% | 0.26 пп | 7.3 пп |

   Для 66% строк \|ROE_2pt − ROE_13pt\| < 0.5 пп; в хвосте выбросы до 217 пп — на банках с резким ростом капитала (bank_20: с 2 млрд до сотен млрд за год, где любая интерпретация «среднего» даёт ROE сотен процентов, и методы расходятся сильно). На стабильных банках выбор метода меняет ROE на десятые доли процентного пункта — результат легко пересчитать под `(begin+end)/2` или 12-точечную схему.

   Распределение \|ΔROE\| vs 13pt асимметричное и для 12-точечного метода: p95 = 2.3 пп при медиане 0.07 пп; хвост сформирован теми же банками с резким изменением капитала (из 52 строк на/выше p95: bank_20 — 22, bank_10 — 11, bank_5 — 11, остальные — единицы). Топ-10 по \|ROE_12pt − ROE_13pt\| — целиком bank_20 (Δ ≈ 18–25 пп на фоне самого ROE 250–385%). Для «нормальных» банков все три метода практически неотличимы.

5. **NIC и NII имеют разный период накопления.** NIC — годовое накопление (YTD), NII — квартальное (QTD). Оба соответствуют тексту ТЗ (разные периоды коэффициентов), но сам факт, что рядом в одном датасете лежат ряды с разной периодикой накопления сильно сбивает с толку
### 3.3. Тест-кейсы

С использованием LLM сгенерированы 3 тест-кейса на краевые значения (подробно — `test_cases.md`):

1. **Equity меняет знак в окне ROE 12m.** Среднее Equity может быть положительным, но наличие отрицательных точек делает ROE бессмысленным. Ожидание: ROE = NaN.
2. **|LLP| > Loans_Total_Net.** Резерв по модулю больше портфеля — ошибка данных. Loans_Gross формально > 0, но LDR завышен. Ожидание: флаг предупреждения.
3. **Пропуск квартальной точки NII.** NII — QTD: каждое значение — накопление внутри своего квартала, независимо от других. Пропуск одной точки не ломает соседние. Ожидание: NaN только для пропущенного квартала.

### 3.4. Как проверять корректность расчётов

1. **Перекрёстная проверка profit_12m.** Rolling-сумма 12 месячных потоков должна совпадать с формулой: `profit_12m(t) = NIC(t) + NIC(1 янв Y(t)) − NIC(t − 12 мес)`. Расхождение > ε указывает на ошибку в восстановлении потоков.
2. **Сверка с альтернативной интерпретацией.** Параллельный расчёт ROE под flow и YTD гипотезами. Если обе версии дают аномальные значения, то проблема в данных.
3. **Диапазонные проверки.** ROE ∈ [−100%, +100%], LoanYield ∈ [0%, 50%], LDR ∈ [0.1, 3.0]. Выход за границы → флаг outlier для ручной проверки.
4. **Монотонность YTD внутри года.** NIC (фев → дек) должен расти монотонно для банка с положительной прибылью. При немонотонности стоит посмотреть на данные.
5. **Точечная ручная сверка.** Выбрать 2–3 банка, вручную из Excel посчитать ROE за один год, сравнить с выводом скрипта

### 3.5. Какие ограничения и проверки должны быть внедрены

**На входе (валидация данных):**
- Проверка типов: все числовые колонки приводятся к float; текстовые значения с единицами нормализуются автоматически
- Проверка знаков: Equity < 0 → блок ROE; Loans < 0 → блок LDR и LoanYield; Client_Deposits ≤ 0 → блок LDR
- Проверка полноты: наличие ≥13 последовательных точек для ROE; наличие NII для квартальных дат
- Проверка |LLP| ≤ |Loans_Net|: резерв не может превышать портфель

**На выходе (валидация коэффициентов):**
- Диапазонные флаги: ROE > |1|, LoanYield > 50%, LDR > 3 или < 0.1
- Outlier-маркировка: значения за пределами диапазона помечаются, но не удаляются — решение за аналитиком
- Логирование: число посчитанных/пропущенных строк по каждому коэффициенту и причина пропуска

## 4. Описание использования LLM

LLM (Claude Opus 4.6) использовалась на следующих этапах:

1. **Формализация методологии** Текстовое описание трёх коэффициентов из ТЗ переведено в исполняемые формулы с явными условиями применимости: длина окна, обработка пропусков, условия на знак Equity/Loans/Deposits, аннуализация. Формализация восстановления месячных потоков из YTD, формула Loans_Gross = Net − LLP, диапазонные флаги outlier.

2. **Интерпретация данных** LLM первоначально предложил flow-гипотезу (то есть помесячный поток без накопления) для обеих потоковых величин и аргументировал её правдоподобной годовой суммой NIC за 2020 и немонотонностью внутри года (14/66 отрицательных diff, что, цитата, «типично для flow и нетипично для YTD»). Изначально я поверил этой интерпретации, но сомнения появились только когда расчёты дали ROE с медианой 87% и максимумом 1217%. Такой результат в реальности практически невозможен в банковском бизнесе. На вопрос «А это не может быть YTD?» LLM предложил ratio-тест (5–10 по всем банкам — диапазон, совместимый с YTD). Для NII LLM изначально сравнивала январь с апрелем (разные периоды YTD), что было логически некорректно; корректным оказалось сравнение Q1 vs H1 внутри одного года, которое опровергло YTD-по-всему-году для NII и позволило LLM интерпретировать данные в соответствии с ТЗ. Итоговый выбор: NIC=YTD, NII=QTD (накопление внутри квартала, см. ниже).

3. **Парсинг и нормализация.** Функция `parse_value` для смешанных единиц (млн/тыс./млрд руб., NBSP как разделитель, запятая как десятичный знак) с регрессионным тестом на 11 кейсах.

4. **Выявление аномалий.** Систематизация типов ошибок (отрицательные балансы, |LLP| > |Net|, текстовые значения с единицами), формализация флагов

5. **Генерация тест-кейсов.** 3 краевых случая (смена знака Equity в окне ROE, |LLP| > Net для LDR, пропуск квартальной точки NII) в `test_ratios.py`

6. **Проверки корректности.** Параллельный расчёт YTD и помесячного потока, диапазонные outlier-флаги, ручная сверка на основе данных по bank_1

### О роли человека

Главная ошибка LLM на этой задаче была не в расчёте и не в выборе формул, а в работе со спецификацией. В ТЗ написано: «прибыль -- потоковая величина, отражается накопленным итогом», «ЧПД отражается накопленным итогом за квартал» — это прямое указание на накопление, т.е. в обоих случаях указание на накопленный итог, но у ЧПД накопленный итог за квартал. LLM эту зависимость не смогла сразу установить и трактовала «накопленный итог» как глобальный YTD для обеих величин. Правильная последовательность работы должна была быть:

1. ТЗ — источник истины по умолчанию
2. Проверить, совместимы ли данные с ТЗ
3. Документировать расхождения как ошибки данных
4. Отклоняться от ТЗ только там, где данные однозначно не допускают иную интерпретацию

LLM видит аномалию в данных и «подгоняет» интерпретацию ТЗ под данные, вместо того чтобы сначала аккуратно прочитать ТЗ

Итог: LLM эффективна для формализации алгоритма расчета коэффициентов, парсинга, генерации тестовых сценариев. На этапах чтения спецификации, математической валидации аргументов и интерпретации обязательно нужен эксперт
