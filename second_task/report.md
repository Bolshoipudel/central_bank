# Отчёт: расчёт финансовых коэффициентов по данным bank_data.xlsx

## 1. Формализованная логика расчёта

### Исходные данные

Файл `bank_data.xlsx`: 20 банков × 73 даты (2020-01-01 … 2026-01-01), помесячно. Колонки:
- Балансовые (срез на 1-е число месяца): `Assets`, `Loans_Total_Net`, `Loans_LLP`, `Equity`, `Client_Deposits`
- Потоковые: `Net_Income_Current` (помесячно), `Net_Interest_Income` (только Jan/Apr/Jul/Oct)

### Обозначения

- `t` — отчётная дата (1-е число месяца)
- `NIC(t)` — Net_Income_Current на дату t
- `NII(t)` — Net_Interest_Income на дату t

### Про потоковые величины

В ТЗ написано, что обе потоковые величины «отражаются накопленным итогом» (YTD). Эмпирическая проверка на данных показала, что однозначно опровергнуть YTD удаётся только для NII; для NIC формальные тесты неоднозначны, и окончательный выбор сделан по реалистичности итоговых коэффициентов.

**NIC — принято как YTD (соответствует ТЗ):**
Ratio-тест `Σ NIC за 12 месяцев / NIC(1 янв следующего года)` даёт 5–10 по всем банкам. Этот тест сам по себе не различает гипотезы: под YTD с ~равными месячными потоками ratio ≈ 6.5; под flow с ростом от года к году (реальный случай, 2020 → 2025 прибыль растёт в ~5×) ratio тоже может быть 7–10. Решающим аргументом стала реалистичность ROE: под YTD медиана 15% (соответствует банковской норме 10–20%), под flow медиана 87% с max 1217% (нереалистично).

**NII — по ТЗ должен быть YTD, но данные это однозначно опровергают:**
Значения внутри года не растут монотонно (bank_1 2024: Q1=17 625 → H1=16 920 — под YTD невозможно). YTD-формула даёт LoanYield ~0.9% (нереалистично), flow-формула — ~7% (соответствует рыночным значениям). Принято: NII используется как flow, расхождение с ТЗ зафиксировано как найденная проблема.

### Восстановление месячной прибыли из YTD

```
profit_month(t) =
  NIC(t),                если month(t) = 2   (январский поток = YTD на 1 фев)
  NIC(t) − NIC(t−1мес),  иначе               (включая t = 1 янв: FY − YTD на 1 дек)
```

NIC(1 янв Y) трактуется как FY(Y−1), т.е. YTD для нового года стартует с 0.

### Коэффициент 1: ROE 12m

```
profit_12m(t) = Σ profit_month(s), s ∈ {t−11, …, t}     # 12 месячных потоков
equity_avg(t) = mean(Equity(s)),   s ∈ {t−12, …, t}     # 13 балансовых точек

ROE_12m(t) = profit_12m(t) / equity_avg(t)
```

Условия расчёта:
- Требуется ≥13 точек истории (окно t−12 … t)
- Если хотя бы одно Equity(s) < 0 в окне → ROE не определён (по ТЗ)
- Если в окне есть пропуски NIC или Equity → ROE не определён

### Коэффициент 2: Доходность кредитного портфеля 3m

Определён только для квартальных дат: t ∈ {1 янв, 1 апр, 1 июл, 1 окт}.

```
loans_avg_Q(t) = mean(Loans_Total_Net(s)),  s — 4 точки квартала

LoanYield_3m(t) = 4 × NII(t) / loans_avg_Q(t)     # аннуализация ×4
```

NII(t) используется как квартальный flow (см. обоснование выше).

Условия расчёта:
- NII(t) определён (не NaN)
- Все Loans_Total_Net в окне ≥ 0
- loans_avg_Q > 0

### Коэффициент 3: LDR (Loans-to-Deposits Ratio)

```
Loans_Gross(t) = Loans_Total_Net(t) − Loans_LLP(t)     # LLP < 0, поэтому Gross > Net

LDR(t) = Loans_Gross(t) / Client_Deposits(t)
```

Условия расчёта:
- Loans_Gross(t) ≥ 0
- Client_Deposits(t) > 0

## 3. Найденные проблемы

### 3.1. Ошибки и аномалии в данных

Эксперт, предоставивший датасет, подтвердил, что `bank_data.xlsx` — синтетические данные, в которых «возможны ошибки». Это объясняет, почему ни одна интерпретация не даёт идеально чистой картины, и оправдывает выбор по эмпирическому критерию (реалистичность итоговых коэффициентов) там, где формальные тесты неоднозначны.

| Проблема | Число строк | Затронутые банки |
|---|---|---|
| Equity < 0 | 172 | bank_3 (72/73), bank_11 (73/73), и др. |
| Loans_Total_Net < 0 | единичные | bank_1 (2023-07) |
| Client_Deposits < 0 | 5 | — |
| Assets < 0 | 3 | — |
| Текстовые значения с единицами | ~30 ячеек | «729 873,3 млн руб.», «… тыс. руб.» |
| Смешанные единицы измерения | — | млн руб. (основные), тыс. руб., млрд руб. |

### 3.2. Неоднозначности в методологии

1. **Семантика NII не соответствует ТЗ.** ТЗ говорит «ЧПД отражается накопленным итогом» (YTD), но в данных NII ведёт себя как квартальный flow: значения внутри года немонотонны, YTD-формула даёт нереалистичный LoanYield ~0.9%. Возможно это ошибка данных, возможно — неточность формулировки ТЗ.

2. **NIC(1 янв Y): FY(Y−1) или ноль нового года?** По значениям (bank_1 2020-01: 7 873, 2021-01: 8 989) — полная годовая прибыль предыдущего года, не сброс. Влияет на восстановление месячных потоков.

3. **Loans Gross: Net − LLP или Net as-is?** ТЗ говорит «кредиты (gross)», но отдельной колонки нет. Принята формула Gross = Net − LLP (LLP отрицательный → Gross > Net). Альтернатива — использовать Net.

4. **Среднее Equity для ROE: 13 точек или 2?** «Среднегодовое значение» можно трактовать как среднее всех помесячных балансов (13 точек) или как (начало + конец) / 2. Принято 13 точек.

5. **NIC и NII имеют разную семантику в одном датасете.** NIC — YTD, NII — flow. Это нетипично для банковской отчётности и может быть артефактом синтетических данных.

### 3.3. Тест-кейсы

С использованием LLM сгенерированы 3 тест-кейса на крайние значения (подробно — `test_cases.md`):

1. **Equity меняет знак в окне ROE 12m.** Среднее Equity может быть положительным, но наличие отрицательных точек делает ROE бессмысленным. Ожидание: ROE = NaN.
2. **|LLP| > Loans_Total_Net.** Резерв по модулю больше портфеля — ошибка данных. Loans_Gross формально > 0, но LDR завышен. Ожидание: флаг предупреждения.
3. **Пропуск квартальной точки NII.** NII — flow, поэтому пропуск одного квартала не ломает соседние. Ожидание: NaN только для пропущенного квартала.

### 3.4. Как проверять корректность расчётов

1. **Перекрёстная проверка profit_12m.** Rolling-сумма 12 месячных потоков должна совпадать с формулой: `profit_12m(t) = NIC(t) + NIC(1 янв Y(t)) − NIC(t − 12 мес)`. Расхождение > ε указывает на ошибку в восстановлении потоков.
2. **Сверка с альтернативной интерпретацией.** Параллельный расчёт ROE под flow и YTD гипотезами. Если обе версии дают аномальные значения — проблема в данных.
3. **Диапазонные проверки.** ROE ∈ [−100%, +100%], LoanYield ∈ [0%, 50%], LDR ∈ [0.1, 3.0]. Выход за границы → флаг outlier для ручной проверки.
4. **Монотонность YTD внутри года.** NIC (фев → дек) должен расти монотонно для банка с положительной прибылью. Нарушение — повод проверить данные.
5. **Точечная ручная сверка.** Выбрать 2–3 банка, вручную из Excel посчитать ROE за один год, сравнить с выходом скрипта.

### 3.5. Какие ограничения и проверки должны быть внедрены

**На входе (валидация данных):**
- Проверка типов: все числовые колонки приводятся к float; текстовые значения с единицами нормализуются автоматически
- Проверка знаков: Equity < 0 → блок ROE; Loans < 0 → блок LDR и LoanYield; Client_Deposits ≤ 0 → блок LDR
- Проверка полноты: наличие ≥13 последовательных точек для ROE; наличие NII для квартальных дат
- Проверка |LLP| ≤ |Loans_Net|: резерв не может превышать портфель

**На выходе (валидация коэффициентов):**
- Диапазонные флаги: ROE > |1|, LoanYield > 50%, LDR > 3 или < 0.1
- Outlier-маркировка: значения за пределами диапазона помечаются, но не удаляются — решение за аналитиком
- Логирование: число посчитанных/пропущенных строк по каждому коэффициенту и причина пропуска

## 4. Описание использования LLM

LLM (Claude) использовался на следующих этапах:

1. **Формализация методологии.** Текстовое описание трёх коэффициентов из ТЗ переведено в исполняемые формулы с явными условиями применимости: длина окна, обработка пропусков, условия на знак Equity/Loans/Deposits, аннуализация. Формализация восстановления месячных потоков из YTD, формула Loans_Gross = Net − LLP, диапазонные флаги outlier.

2. **Определение семантики данных.** LLM первоначально принял flow-гипотезу для обеих потоковых величин и аргументировал её правдоподобной годовой суммой NIC за 2020 (~67k, похоже на реальную прибыль) и немонотонностью внутри года (14/66 отрицательных diff, что «типично для flow и нетипично для YTD»). Я подхватил этот вывод и даже транслировал его эксперту как свою гипотезу. Сомнения появились только когда расчёт дал ROE с медианой 87% и максимумом 1217% — результат, физически несовместимый с банковским бизнесом. На вопрос «А это не может быть YTD?» LLM предложил ratio-тест (5–10 по всем банкам — диапазон, совместимый с YTD). Для NII LLM изначально сравнивал январь с апрелем (разные периоды YTD), что было логически некорректно; корректным оказалось сравнение Q1 vs H1 внутри одного года, которое однозначно опровергло YTD для NII. Решающим аргументом для итогового выбора (NIC=YTD, NII=flow) стала реалистичность полученных коэффициентов: под YTD медиана ROE 15% (норма), под flow 87% (мусор). Подтверждение эксперта о синтетичности данных с возможными ошибками легитимизировало эмпирический критерий.

3. **Парсинг и нормализация.** Функция `parse_value` для смешанных единиц (млн/тыс./млрд руб., NBSP как разделитель, запятая как десятичный знак) с регрессионным тестом на 11 кейсах.

4. **Выявление аномалий.** Систематизация типов ошибок (отрицательные балансы, |LLP| > |Net|, текстовые значения с единицами), формализация флагов.

5. **Генерация тест-кейсов.** 3 краевых случая (смена знака Equity в окне ROE, |LLP| > Net для LDR, пропуск квартальной точки NII), реализованы как исполняемые ассерты в `test_ratios.py`.

6. **Проверки корректности.** Параллельный расчёт YTD и flow как control-индикатор (ratio в summary), диапазонные outlier-флаги, ручная сверка rolling-логики на bank_1.

### Ключевой вывод о роли человека

Главная ошибка LLM на этой задаче была не в расчёте и не в выборе формул, а в работе со спецификацией. ТЗ буквально говорит: «прибыль отражается накопленным итогом», «ЧПД отражается накопленным итогом» — это прямое указание на YTD. Правильная последовательность работы должна была быть:

1. ТЗ — источник истины по умолчанию.
2. Проверить, совместимы ли данные с ТЗ.
3. Документировать расхождения как ошибки данных.
4. Отклоняться от ТЗ только там, где данные делают её **логически невозможной**, а не просто «странноватой».

Вместо этого LLM увидел аномалии в данных (падения внутри года, немонотонность) и трактовал их как опровержение ТЗ. Это перевернутая приоритизация: спецификация отбрасывается из-за шумной реализации. Я подхватил этот вывод без проверки, довёл его до эксперта и вернулся к правильной интерпретации только после того, как расчёт под flow дал ROE 87% — число, невозможное для банковского сектора.

Для NII итоговое отклонение от ТЗ (flow вместо YTD) оправдано: H1 < Q1 делает YTD логически невозможной, а не просто неудобной. Для NIC никакой логической невозможности не было — данные совместимы с YTD при допущении ошибок в отдельных месяцах, и правильный ответ совпадал с буквальной трактовкой ТЗ.

Отдельно: первоначальная формулировка ratio-теста в отчёте содержала математическую неточность («при flow ожидалось бы ≈1», хотя под flow ratio ≈ 12 при равных потоках и 7–10 при росте). Это пример того, как LLM-сгенерированное обоснование может выглядеть убедительно на уровне изложения, но не выдерживать буквальной проверки — и я это пропустил.

Без конкретных вопросов «А это не может быть YTD?» и «А почему у накопительного так быть не может?» расчёт ушёл бы с ошибкой масштаба ×5–6. Самая полезная моя роль в этом проекте — не генерация идей, а настойчивое оспаривание того, что выдаёт LLM, особенно когда выход выглядит согласованным и аргументированным.

Итог: LLM эффективен для формализации, парсинга, автоматизации расчётов и генерации тестовых сценариев. На этапах чтения спецификации, математической валидации аргументов и интерпретации неоднозначных данных ценность человеческого контроля не во «вспомогательной проверке», а в удержании приоритета «ТЗ > данные > догадки LLM».
